Game Design

The battle loop, the math behind it, and progression.

Core loop: independent attack cycles, not shared rounds

There is no shared "round" that both sides wait on. The player and the monster each run their own attack cycle, continuously and independently, for the whole battle:

  • Player cycle: get a prompt + time limit → type it (backspace freely) → attempt submits (see Submitting an attack) → hit or miss resolves immediately → the very next prompt appears right away. No waiting on the monster.
  • Monster cycle: at the same time, independently, the monster is visibly "typing" its own unrelated prompt against its own clock (see Monster typing) → it hits or misses on its own schedule → it immediately starts its next attack.

Both cycles run in parallel for as long as the battle lasts. A fast, clean typist might land three hits before the monster finishes its first attack; a slow monster might still be correcting a typo while the player is already three prompts ahead. Battle ends the instant either HP bar hits 0 — whichever cycle gets there first.

Rounds are per-side, not shared. A shared/lockstep round means the faster side sits idle waiting on the slower one every single exchange — dead time, and it caps the whole battle's pace at the slower typist. Independent cycles keep the player continuously typing (no idle waiting) and make the pace of the fight an emergent result of both sides' actual speed, which reads as a much more dynamic battle.

Submitting an attack

Backspacing is always allowed — there's no separate accuracy penalty for using it (see Damage: going back to fix a mistake just costs time, which the speed bonus already captures). What matters is what happens when the player presses Return:

  • The typed string must be exactly correct to land the hit. Return is the only way to submit — there's no auto-submit the instant length matches, since prompts may contain in-line line breaks.
  • Return only counts as a submit attempt once the typed input is the same length as the target text. If input is shorter than the target, Return is treated as a literal character (so a prompt with an inline line break types unambiguously).
  • Once length matches and Return is pressed: exact match → hit lands; anything else (even one wrong character) → the attack misses. Either way, the next prompt starts immediately — there's no re-attempt on the same prompt.
Submitting incorrect text is simply not allowed to land a hit — there's no partial credit for "close enough," and no accuracy multiplier softening it. This keeps the mental model simple ("type it right, or it doesn't count") and pushes all of the skill expression into speed, which is easier for a 10-year-old to reason about than a fuzzy accuracy score.

Damage

Base shape, tunable via a config object rather than hardcoded:

damage = (sum of weapon-die rolls + weaponAbilityMod) * damageScale
       * lengthFactor(text)
       * speedBonus(timeUsed, timeLimit)
       * powerUpMultiplier
       * tierGatePenalty
       * fumbleDamageMultiplier
  • weapon die + ability mod — one roll of the equipped weapon's damage die (a Dagger's d4 up to a Greataxe's d12; see Weapons), plus the modifier of whichever ability governs that weapon (STR for a melee weapon, DEX for a finesse one, INT for the Wizard's Wand). A crit doesn't multiply the result — it rolls the die again and sums both: 2 dice normally, 3 for the Wizard's Arcane Mind class feature. damageScale is the one config constant (config/abilities.ts) that keeps dice-rolled hits in the same ballpark the old flat baseDamage used to occupy — see Abilities.
  • lengthFactor — unchanged in shape: roughly 1 + (charCount - referenceLength) / referenceLength, so a longer prompt swings harder — now the payoff for leveling INT high enough to be served longer text (see Abilities), not a skill-tree unlock. Clamped to a small floor (Math.max(0.25, …)) so a very short prompt well under referenceLength can't drive damage to zero or negative.
  • speedBonus — unchanged: multiplier for finishing well under the time limit vs. barely making it. There is no separate accuracy factor: a typo already costs time (notice it, backspace, retype), so sloppy typing is automatically punished by a worse speedBonus rather than by a second penalty.
  • Crit chance — today, still a flat combat.criticalChance per swing, exactly as in M0. DEX's crit-chance bonus and a weapon's crit range are both computed (see Abilities / Weapons) but not yet read by the crit roll itself — a known pending wiring gap, not a design decision; see Codebase Architecture.
  • powerUpMultiplier — 1.0 normally; a consumable like Bull's Strength or Elixir of Might (see Consumables) raises it for a fight or a dungeon. Stacks multiplicatively with a crit if both happen to land on the same swing.
  • tierGatePenalty — 1.0 when the text tier the player is actually served covers this fight's target tier; otherwise (servedTier / targetTier) ** 2. The served tier now comes from the encounter d20's band, capped by the player's INT-driven tier cap — not a monster's own textTier the way M2's Wordsmith gate worked (the monster's textTier still governs only the line the monster itself types).
  • fumbleDamageMultiplier — 1.0 normally; a fumbled encounter roll (natural 1) caps every hit that fight at ×0.75 and suppresses crits outright — see the encounter d20.

The monster's damage uses the same shape minus the dice/crit/power-up/tier-gate pieces (all player-only concepts) — a flat combat.baseDamage × lengthFactor × speedBonus — so difficulty is tuned by monster stats, not a separate rulebook.

Time limit is derived from text length ÷ an assumed baseline typing speed (expressed in WPM — the same unit the monsters use — and converted to chars/sec via the standard 5 chars/word) rather than a flat number, so an INT-driven jump in served text tier doesn't accidentally also make attacks impossibly short. That baseline WPM is the actual difficulty knob: 15 WPM is a realistic beginner rate for the target player, so assuming a baseline at or a touch below his real speed keeps early fights generous. The game then gets harder mainly through longer text and tougher monsters; WIS's flat ms bonus effectively lowers the assumed baseline (more time) as it levels, keeping the clock fair rather than punishing.
Note: the divisor is a typing speed, not a raw chars/sec figure — a baseline of 15 WPM (≈ 1.25 chars/sec), not 15 chars/sec (≈ 180 WPM, pro-typist speed, which would make anything past the 3s floor unplayable for a 10-year-old). Tuning this baseline is a core job of Roadmap M0.

Monster "typing"

The monster's text should actually appear on screen character by character, in real time, typos and all — not an abstract progress bar. That's what makes it read as "the monster is racing you" instead of a hidden dice roll. Each monster has three per-monster stats driving the simulation, plus one shared tuning constant:

  • wpm — mean typing speed. Converted to an average per-character interval, then each character's actual interval is sampled from a distribution around that mean (e.g. a gamma/normal distribution), so the typing has natural, uneven pacing rather than a metronome. The spread of that distribution comes from variance — a single shared constant for v1, not tuned per monster (see roster, which has no per-monster variance column for that reason), promotable to a per-monster stat later if some monster should feel distinctly more erratic than others.
  • accuracy — chance, rolled per character, that the monster types the wrong character instead of the correct one.
  • attention — mean seconds before the monster notices a mistake, not a per-character roll. This is deliberately time-based rather than character-based: a fast typist and a slow typist with the same attention value should take about the same wall-clock time to notice something's wrong — which means the fast one has already banged out more (wrong) characters in that window, and has more to undo. Speed alone makes mistakes costlier; attention is a separate stat for how quickly a monster catches on regardless of its speed.
for each character in monster.prompt:
  interval = sample(mean = 60000 / (wpm * avgWordLength), variance = monster.variance)
  wait(interval)
  typed = roll(monster.accuracy) ? correctChar : wrongChar

  if typed is wrong and not already correcting:
    mistakeStartTime = now
    noticeDelay = sample(mean = monster.attention, variance = ...) // seconds
    correcting = true

  if correcting and (now - mistakeStartTime) >= noticeDelay:
    backspace back to the mistake, retype correctly (costs the time spent typing past it)
    correcting = false
The wait(interval) above is illustrative — you can't block JavaScript's single thread. The real implementation is tick-driven: one loop advances the simulation by a dt (elapsed milliseconds) each frame, and the monster accumulates time toward its next character and toward its noticeDelay. Feeding that loop an injectable clock and a seedable RNG is what makes the whole race reproducible and unit-testable — see the M0 implementation plan.

The attack succeeds if the monster reaches a fully correct string within its time limit, same rule as the player. The monster's time limit is its own expected typing time × a slack factor (~1.75), not the player's formula — a clean, attentive monster finishes comfortably, but a sloppy or slow-to-notice one can run its clock out mid-correction and whiff the attack. (Reusing the player's short budget would leave slow monsters like the Slime unable to ever land a hit.) That slack is a per-battle tuning knob. Because attention is measured in seconds, a monster that's both fast and slow to notice (high wpm, high attention-seconds) is the worst combination — it'll type a wall of extra wrong characters before catching on. That's why stronger/later monsters raise wpm and attention together (see the roster): fast and quick to self-correct, rather than fast and sloppy. A monster with low accuracy and slow attention will visibly flub words and scramble to fix them — exactly the kind of race the player should be able to watch and react to (e.g. holding a power-up because the monster is clearly about to whiff).

Progression: coins, XP, leveling

After a won battle: coins (spend at the Shop), XP (a running total that crosses 5e-style level thresholds automatically — see Leveling — no longer a spendable currency), and a chance at loot.

This is a per-battle system, separate from hearts — the run-level lives system that governs a whole dungeon attempt (lose all hearts, get sent back to the Inn and lose the dungeon's progress). Hearts, the world map, and dungeons themselves are a big enough system to get their own doc — see the M2 scope doc.

The six abilities

M3 replaces the M2 skill tree with six D&D-style ability scores. Every score has a modifier via the standard formula floor((score - 10) / 2) — 8 → -1, 10/11 → 0, 15 → +2, 20 → +5 — and every ability-driven number in the game reads off that one modifier, not the raw score:

Leveling & classes

A real 5e-shaped leveling table replaces the skill tree's purchased nodes: 5e's own XP-threshold curve (0, 300, 900, 2700, 6500 …), a proficiency bonus that climbs +2 → +6 with level and folds into every encounter d20, and an Ability Score Improvement (2 points to spend, e.g. +2 to one ability or +1 to two) banked at levels 4, 8, 12, 16, and 19 — spent at the Inn. Max HP is derived at read time from the character's class hit die × level (never stored, so it can't drift), and max hearts (the per-run lives, see M2 scope) now come from level milestones instead of a purchased Endurance node.

Four launch classes, each with one always-on feature:

Every per-point magnitude above (the damage %, the ms bonus, the tier steps, the class-feature numbers) is a Story 13 placeholder landed by theory, not feel — corralled into config/abilities.ts / config/leveling.ts / config/classes.ts so the real playtest-driven numbers pass (M5) is a values edit, not a code change.

The encounter d20

Every fight opens with a d20 roll (+ proficiency bonus + any item bonus, e.g. a Luckstone) before the clock starts, and the total picks which of the dungeon's three prompt-difficulty bands the player types this fight:

TotalResult
natural 1Fumble — low band, no crits, every hit capped at ×0.75 damage this fight
2–7Low band — the bottom of the dungeon's text-tier range
8–13Mid band — the range's rounded midpoint
14–19High band — the top of the range
natural 20Inspired — high band, and the first landed hit is a guaranteed crit

The band's target tier is then capped by the player's INT-driven tier cap to produce the fight's actual served tier, which is what feeds tierGatePenalty — the same squared-ratio math as before M3, just sourced from the d20's band instead of a monster's own text tier. A Rogue's Cunning rolls this d20 twice and keeps the higher (advantage); a Guidance potion grants the same for one fight.

Weapons

Eight launch weapons — Dagger, Shortsword, Longsword, Rapier, Wand, Warhammer, Longbow, Greataxe — each with a damage die (d4 up to d12), a governing ability, a 5e-style crit range, and a price. Bought at the Shop, equipped one at a time from the Inn's Armory; each class starts with one for free. The Greataxe (d12, the hardest-hitting) taxes the typing time budget as its risk/reward tradeoff — see Damage.

Consumables & the Shop

Ten consumable items — potions, elixirs, oils — bought with coins at the Shop and carried forever until used. Each grants a buff lasting one fight, several fights, or the rest of the dungeon (never persisted past the run, same rule as hearts): instant healing, a power-up-style damage multiplier (Bull's Strength, Elixir of Might), extra typing time (Potion of Speed), encounter-roll help (Guidance's advantage, a Luckstone's flat bonus), a crit-chance/damage buff (Oil of Sharpness), a temporary INT tier-cap bump (Elixir of Intellect), and fumble immunity plus bonus HP (Potion of Heroism). The player chooses when to use one from the dungeon map's Bag — a tap/click before a fight, not something that interrupts typing. Prices and magnitudes are Story 13/M5 placeholders like everything else ability-driven.
