State: no Redux, but three distinct homes

"State in React components" is right for the save file and the menus, but a battle is a real-time simulation, not a form — the monster's live typing, both HP bars, and both countdowns update many times a second. Pushing that through scattered useState/useEffect tangles the combat math with rendering and makes "do the numbers feel right?" impossible to test in isolation. The fix is not a state library; it's putting each kind of state where it belongs:

KindExampleHome
Simulation / battlemonster's typed string, HP bars, countdownsA framework-agnostic engine (a factory function per no-class) that owns the state and advances via tick(dt); React reads it through useSyncExternalStore
Persistent / savecoins, the character (class/level/XP/abilities), equipped weapon + inventoryReact Context + useReducer — cross-component and low-frequency, the textbook Context case (arrives in M2)
Ephemeral UIthe input box text, which screen is upLocal useState in the component
Redux/Zustand really would be overhead — but the reason isn't "the app is small," it's that none of these three problems is the one Redux solves. Context re-renders every consumer on every change (fine for a save file that changes once a battle, pathological for state ticking 60×/sec), which is exactly why the battle lives in an external-store engine instead. Keeping that engine free of React imports is also what lets M0 test the combat math headlessly and run a balance simulator over it. See the M0 implementation plan for the file layout that draws this boundary.

Code style: ESLint + Prettier baseline

SettingValue
ESLint baseeslint-config-airbnb + airbnb-typescript (TS support) + eslint-plugin-react / react-hooks
Line length100 characters
Semicolonsnone — ASI (Automatic Semicolon Insertion)
Indentation2 spaces, tabs disallowed
FormatterPrettier, printWidth: 100, semi: false, tabWidth: 2, useTabs: false; eslint-config-prettier turns off any ESLint stylistic rule that would fight it
Prettier is what makes semicolon-free code actually safe, not just shorter. The classic ASI footgun is a line starting with (, [, a backtick, +, or - silently merging with the line above and changing meaning. Prettier detects exactly this and inserts a disambiguating leading ; automatically — so ASI safety comes from letting the formatter run, not from writing careful code by hand.

Two custom syntax bans

Enforced via ESLint's no-restricted-syntax (no extra plugin needed — it targets AST node types directly):

  • no-function — ban FunctionDeclaration and FunctionExpression; arrow functions only. Object method shorthand ({ attack() {...} }) doesn't use the literal function keyword, so it's unaffected and stays the idiomatic way to put a method on a plain object.
  • no-class — ban ClassDeclaration and ClassExpression; use factory functions (a function returning a plain object, closing over any private state) instead of instantiated classes. This only bans authoring classes in our own code — using a class from a library (e.g. whatever idb hands back, new Map(), new Error()) is unaffected.

Things worth knowing going in

  • Hoisting changes. Function declarations hoist; arrow-function consts don't. Any code that assumed "used before defined further down the file" (e.g. mutually recursive helpers) needs reordering under no-function.
  • No generators, no constructor-based patterns. Arrow functions can't be function* generators and can't be used with new. Not expected to matter here, but if the monster-typing character stream (Monster typing) ever wants a generator, that's a spot to revisit.
  • Error subclassing needs a factory instead. The common class MyError extends Error {} pattern is unavailable under no-class; use Object.assign(new Error(message), { code }) or similar instead.
  • Costs nothing on the React side. This project was already going to be function components + hooks, never class components — no-class doesn't fight the framework. eslint-plugin-react-hooks (rules-of-hooks, exhaustive-deps) belongs in the baseline since it catches real hook bugs, not just style.
  • Airbnb's import rules and Vite/TS don't always agree. import/extensions and import/no-unresolved are tuned for older Node/webpack resolution and can misfire on Vite + TS path resolution — expect a resolver setting or a couple of targeted overrides, not a wholesale disable.
  • no-console. Airbnb bans it outright. Worth relaxing to warn-only (or allowing console.warn/console.error) rather than off — a solo side project benefits from leaving some breadcrumbs in, but shouldn't do so silently.
  • Enforce it, don't just declare it. Wire ESLint + Prettier into a pre-commit hook (husky + lint-staged) and add a lint step to the GitHub Actions build (see Getting it onto the Chromebook) so a lint failure blocks the deploy — otherwise a "baseline" is just a suggestion nobody's enforcing.

Save data & persistence

One JSON-shaped record, versioned so future shape changes can migrate it:

{
  "version": 3,
  "character": {
    "name": "Aria",
    "class": "fighter",
    "level": 3,
    "xp": 900,
    "abilities": { "str": 15, "dex": 12, "con": 14, "int": 10, "wis": 10, "cha": 8 },
    "pendingAsi": 0
  },
  "coins": 120,
  "equippedWeapon": "longsword",
  "inventory": {
    "weapons": ["dagger", "longsword"],
    "consumables": { "potion-healing": 2, "luckstone": 1, "...": 0 }
  },
  "hearts": { "max": 2 },
  "highestUnlockedTier": 3,
  "monstersDefeated": ["slime", "goblin"],
  "stats": { "battlesWon": 7, "battlesLost": 2, "bestWpm": 28 }
}
character replaced the M2 skillTree/xp fields in M3 (see the M3 scope doc) — null is the "uncreated" marker that routes a fresh or freshly-migrated save through character creation before the world map. equippedWeapon and inventory are new alongside it; XP now lives on the character and crosses level thresholds automatically instead of being a spendable currency. hearts/highestUnlockedTier (added in M2, see the M2 scope doc) carry over unchanged. Deliberately still not in this record: the in-progress dungeon graph, cleared nodes, hearts remaining mid-run, or any active consumable buff — that state is meant to be thrown away on death/leave/re-visit by design, so persisting it would fight the game's own rules rather than support them.

Stored in IndexedDB rather than localStorage — via a thin wrapper like idb so the app code never touches IndexedDB's raw callback API directly. Access goes through a small interface, not scattered get/set calls: 

interface SaveStore {
  load(): Promise<SaveData | undefined>;
  save(data: SaveData): Promise<void>;
}
IndexedDB over localStorage, specifically for the migration path: its API is already async and works with structured objects, which is exactly the shape a future network call (fetch to a real backend) would have too. If a cloud DB is ever wanted — cross-device sync, a second kid's profile, anything — swapping the SaveStore implementation from an IndexedDB version to a networked one is a localized change behind that one interface, with no rewrite of call sites. localStorage's synchronous get/set API doesn't have that shape, so every caller would need to change, not just the storage layer.
Still no backend/cloud sync in v1 — this is purely about leaving the door open cheaply, not building it now. See Roadmap: this is a post-V1 concern, not a blocker for any milestone.

Data model for content

Monsters and text banks are plain data files (JSON or .ts const exports), not hardcoded in components — see Content Plan. This keeps "add a monster" or "add ten lines to tier 3" a content edit, not a code change.

Getting it onto the Chromebook

  1. Host it as a static site. Push the repo to GitHub, deploy via GitHub Pages — free, no new account to create, auto-deploys on every push to main via a GitHub Actions workflow. Result: a stable https://… URL.
  2. Open it in Chrome on the Chromebook and use "Install app" / "Add to Home screen" from the browser menu — because it's a PWA, this pins a real icon to the shelf/app launcher and it opens full-screen like a native app.
  3. Offline support comes for free from the PWA service worker once assets are cached — nice-to-have, not required for v1 since the Chromebook is normally online.
  4. No app store, no review process, no cost. Updates ship the instant you push to main — refreshing the installed app picks up the new build.
Penciled in: GitHub Pages, since a GitHub account already exists — no new sign-up, and it's free. It needs one extra piece Vercel/Netlify don't: a small GitHub Actions workflow to build the Vite app and publish dist/ to Pages (a standard, copy-pasteable workflow — a few lines of YAML, not custom infra). Easy to switch to Vercel or Netlify later (see the comparison below) if the GitHub Actions build step ever becomes annoying — nothing about the app itself is host-specific.

Any of these can later have a real custom domain attached (e.g. dungeonsandtyping.com) if ever wanted — a few dollars a year for the domain, a few minutes of DNS setup, no code changes. Not needed to start; the free subdomain is a perfectly stable, bookmarkable/installable URL on its own.

Worth knowing: that URL is public on the open internet — not listed anywhere or discoverable without the exact link, but not access-controlled either. Fine for this project (it's a game, no personal data beyond a local save file), so no login wall is planned. GitHub Pages itself has no built-in access control on the free tier (unlike Vercel/Netlify, which offer a one-click "password protect this deployment" toggle) — if that ever feels wrong, that toggle is one reason to switch hosts, rather than building real auth.

Non-goals (for now)

  • Multiplayer / PvP
  • Accounts, login, or cloud sync — deliberately left possible via the IndexedDB SaveStore interface above, but not built for v1
  • Mobile app store packaging (PWA covers "icon on the device")
  • Monetization of any kind
