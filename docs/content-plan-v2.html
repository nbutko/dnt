Content Plan v2

A gentler 14-tier ladder, and a pipeline to fill it.

Supersedes the tiering & sourcing sections of Content Plan v1. Two big changes: a gentler, longer ladder (10 tiers → 14, so the early ramp introduces one new thing at a time instead of jumping home-row → full sentences in two steps), and a real generation method — v1 shipped a handful of hand-written lines per tier (~8 per section, measured — nowhere near enough), v2 defines a repeatable pipeline that produces a large, on-theme, almost non-repeating pool, generated offline and shipped as static JSON.

Every number below was prototyped and measured on the Swamp (Dungeon 6), not projected. Prototype artifacts live under content-pipeline/.

1. The 14-tier ladder

Design rule: each tier adds exactly one new demand — new keys, or more length, or one new character class. Character sets are cumulative from T5 upward (each tier is a superset of the one below). Lengths are order-of-magnitude guidelines, not hard gates.

Home row (1) → left hand (2) → right hand (3) is deliberate. Right comes after left because with the left hand busy the right is free to reach Backspace reliably — a 10-year-old fixing mistakes needs that. Only at (4) do both hands work together, in short bursts, before length arrives at (5). And the complexity ceiling is reached at T9: tiers 10–14 add nothing to the character set — they are a pure length/stamina ramp, which makes them a different content shape (passages, not lines).

2. Dungeon → tier mapping

11 dungeons, 14 tiers. Dungeon N serves regular tiers [N, N+1, N+2]; its boss serves N+3. The encounter d20's band picks within the range (low → N, mid → N+1, high → N+2); the boss is forced to the top+1.

  • config/dungeon-tiers.ts re-mapped to the [N, N+2] ranges above.

The one new plumbing need: content is indexed by difficulty (tier) and theme (dungeon + section). To let a section draw its own themed pool, FightEncounter (state/battle-store.ts) must carry the dungeon id + section alongside textTierRange.

3. The content-generation method

3.0 Non-negotiable principle: generate offline, ship static JSON

No LLM and no generator runs on the kid's Chromebook. Every technique below is an authoring-time batch; its output is curated JSON bundled at build time — same as v1, just far more of it. This keeps the "no backend, no accounts, works offline" invariant and lets us quality-gate every line before it ships.

3.1 The corpus: a hidden campaign per dungeon

Each dungeon has a hidden 5-part D&D campaign built from its habitat + monster roster, run by the standing cast (§4) with pre-rolled dice (dice.json, 1000 each of d4/d6/d8/d10/d20, consumed in order — inconvenient rolls are written into the story, not re-rolled; the chaos is a feature). The dungeon's five structural sections map to the five campaign parts, so the story advances as the kid goes deeper. The campaign is the source of voice and themed vocabulary — not the bulk itself (hand/LLM-authored prose tops out at ~23 usable lines per section; it feeds the engine below).

3.2 The engine: seed-and-compose

We tested three approaches and kept the third:

  • Raw harvest (slice lines straight from transcripts): too scarce (~8 T6/section), and dialogue can't be concatenated into long-form (chopping across speakers = word-salad).
  • Templates (slot-filling from word banks): unlimited volume (~6,800 T6 lines from a 50-word seed) but ~⅓ are broken (the planks waits, the torch stirs across the boardwalk) and even the clean ones feel mechanical and boring. Reserved only for the drill tiers (§3.3), never for prose.
  • Seed-and-compose ✅ — random words drawn from the voiced corpus, handed to the LLM as inspiration to write fresh sentences. Keeps the corpus's flavor + vocabulary; the LLM keeps it grammatical and alive; the tier target keeps it legal.
ModeSeedOutputFeedsMeasured yield
compose-short4 words / 3 adjacent lines4 tier-targeted sentences (one each T6–9)T6–9 bulk~4 clean prompts/draw
compose-long20 words / 15 linesone ~2000-char coherent passageT10–14, and re-harvests to T6–91×T13 + 2×T12 + 6×T11 + 9×T10 from a single 1824-char run

Draws are cheap, so seed quality is handled by over-sampling: filter stopwords + weak words (interjections/contractions/generic verbs), require ≥3 content words per draw (re-draw otherwise), and bias the random start toward narration-heavy lines (they seed denser than quip-heavy dialogue). Raw strong-seed rate ≈ 40%; you just run more draws.

3.3 Drill tiers (T1–5): constrained generation, not harvest

Prose can't produce home-row-only or single-hand words — those are keyboard constraints that fight natural language. T1–5 come from a constrained word-list generator: filter a dictionary to each tier's legal key set, compose short phrases, and deliberately seed rare letters (j/q/z/x/v) so they get practiced. This is the one place templates are the right tool (and the only source for T1–3). It only matters for Dungeons 1–3, where these tiers appear.

3.4 The lint (runs on every generated batch)

  1. Tier-legality — character set + length window per tier (already prototyped).
  2. Dedup — drop near-duplicates.
  3. Coverage — assert each (tier × section) cell hits its target count and its required character classes (T8 every line has a digit; T9 has shifted symbols) and enough letter diversity. Gaps get flagged and back-filled, not shipped silently.

3.5 Volume target

Targets decrease as tiers lengthen, because longer prompts are consumed far fewer per battle (a T6 fight burns ~8–12 short prompts; a T14 fight is only ~2–3 of the 2000-char walls) and the encounter d20 splits each stage's fights across its three tiers, so a single (stage × tier) cell is drawn only ~one battle's worth per playthrough:

Tier bandTargetKeyed by~fresh replays
T1–5 (drills)≥100 (floor)flat, per tierfree tier — overshoot on T4–5; T1–3 cap near their vocabulary ceiling
T6–10~50per (dungeon-stage × tier) cell~5–8
T11–12~30per cell~7
T13–14~20per cell~7–8

The shrinking pools track the shrinking consumption, so every band holds ~5–8 fresh replays before the pool cycles — and within any single sitting the repeat rate is near zero everywhere (you draw ~10 from a 50-pool, ~2–3 from a 20-pool). compose-short at ~4/draw fills a T6–9 cell in ~13 good draws; compose-long fills a length-tier cell in a run or two. The batch is one-time and offline, so overshoot where it's cheap (the drill tiers especially).

4. The cast (draft — tweak freely)

Standing ensemble reused across every dungeon. Character handles, not real-person names: each name carries its own flavor and lets the group be varied without it reading as tokenism. Voices are tuned so different members feed different tier bands.

Mab — the GM (narrator)

Runs the table; warm, vivid, a little wicked — lives for a cliffhanger and a well-timed "…roll me a save." Paints the scene in short sensory strokes, then drops the mechanics like a trap door. Feeds: scene description → T6/T7; stat blocks & calls for rolls → T8/T9. Also the primary source of long-form narration (T10–14).

Bront — the Fighter (STR/CON · longsword · Second Wind)

Ex-caravan guard. Calm, immovable, allergic to drama; has survived worse and doesn't need you to know it. Voice: terse deadpan, four-word sentences, understatement as a personality — "That's a big lizard." Feeds: T6 goldmine (short plain declaratives).

Quill — the Wizard (INT/WIS · wand · arcane mind)

Youngest at the table, oldest soul, carries three notebooks. A rules-lawyer who wants to be correct more than brave. Voice: precise, fussy, over-explains, cites the numbers — "It has resistance to bludgeoning; the club was a mistake, statistically." Feeds: T8/T9 goldmine (HP, gold, 2d6+3, DCs, percentages).

Sable — the Rogue (DEX/CHA · dagger · cunning)

Joined "for the loot," stays for the party; trusts locks more than people, already counting treasure that isn't hers yet. Voice: sly, sarcastic, deflects with questions and asides — "How much do we think the teeth are worth?" Feeds: T9 (the ! and ? of a heist gone sideways).

Fife — the Bard (CHA/WIS · rapier · silver tongue)

Self-appointed chronicler of the party's legend, most of which he's inventing in real time; would rather die than miss a rhyme. Voice: theatrical, florid, pun-forward, exclamation-heavy, narrates his own heroics in third person — "Fife the Undaunted advances — mostly!" Feeds: T7/T9 (flourish, dashes, exclamations).

6. Status & build order

Proven on the Swamp: the 14-tier ladder shape and the D&D-mechanics → free T8/T9 numbers/symbols; compose-short (~4 clean prompts/draw) and compose-long (fills a length-tier cell/run); the tier-legality lint.

Not yet proven / to do: a verified clean T14 (compose-long targeted at ~2300 chars); the drill-tier generator (T1–5); the coverage lint and dedup at batch scale; the section→pool engine plumbing.
Suggested sequence: (1) lock cast + ladder → (2) build the seed-and-compose batch + lint as a runnable tool → (3) generate the full Swamp pool and playtest → (4) build the drill generator → (5) do the engine wiring (TextTier 1..14, dungeon remap, boss tier, section pools) → (6) roll the pipeline across the remaining 10 dungeons.

7. Implementation — generating it cheaply

Done well this is a one-time ~1M token job for the whole game's content library. Done badly it's tens of millions. The difference is one principle:

Scripts do all the reading; agents only ever compose. Deterministic TypeScript (run with npx tsx, matching the codebase) does the corpus-reading (draws), the legality/dedup/coverage checks, and the T1–5 drills — all at zero token cost. LLM tokens are spent only on the compose step, and each compose agent sees a tiny slice of context (a handful of seed-word bundles), never a whole transcript, plan, or codebase.

7.1 How much to generate

Content is keyed by (dungeon × section × tier). Two bands, very different economics:

BandSourceAppears inTargetToken cost
T1–5 (drills)deterministic word-list generatordungeons 1–3≥100 / tier (flat), overshoot on T4–5~zero (no LLM)
T6–10 (short prose)compose-short (+ compose-long for T10)dungeons 3–10~50 / cellLLM

That's roughly ~100–140 prose cells to fill → on the order of ~4,500 short prompts + ~800 long passages (and each long passage re-slices into more short prompts). The drill tiers add ~500+ phrases for free. Enough that a kid won't see a repeat for a very long time.

At the §3.5 volume targets (~50 / T6–10 cell, ~30 / T11–12, ~20 / T13–14):

About a million tokens for every dungeon's content, generated once and shipped as static JSON forever — it's an authoring cost, not a per-play cost. This is a projection; step 7.5 measures the real number on the Swamp before we trust it.

7.4 How this blows up (the anti-patterns to avoid)

  • One monolithic agent generating everything: context grows unbounded, every call re-reads the accumulated history, and quality degrades as it fills. 50–100× the cost.
  • One agent per prompt: thousands of calls, each paying fixed instruction/overhead tokens to emit one line.
  • Agents that read the corpus/plan/codebase each call: the single biggest input-token sink. The draw & lint scripts exist precisely so agents never do this.
  • Regenerating a whole cell on a coverage miss instead of dispatching a small top-up for just the shortfall.
  • Recomposing cells that already passed: draws are free to re-run, composes are not — never recompose a good cell.

7.5 Execution — explicit spawn plan

Run it as a single deterministic workflow script (the L0 orchestrator). Two rules make the whole thing safe:

The orchestrator is a script, not an LLM; agents are leaf workers, exactly two tiers deep. L0 is plain JS — it accumulates no token context as it tracks the 143-cell manifest, so there is no reason to "offload" its bookkeeping into nested agents. Context stays small because the agents (L1 showrunner, L2 filler) each see a tiny slice — not because the tree is deep. No agent spawns another agent (that would be the monolith anti-pattern re-introduced one level down). Fan-out lives in the script; composing lives in the leaves.

L0 — orchestrator (the workflow script)

Spawns, in this order:

  1. 1 drill generator (deterministic TS, T1–5) — independent and free; kick it off first and forget it.
  2. 11 dungeon pipelines, dispatched as a pipeline (no barrier between dungeons): dungeon 1 can be composing while dungeon 7's showrunner is still writing. Each pipeline is the four stages below.
  3. As each dungeon returns, write its JSON pools; emit a final coverage report.

Per-dungeon pipeline (11 in flight, throttled by the concurrency cap)

  1. Stage 1 — Showrunner: 1 agent, serial. Writes the dungeon's 5-section transcript. Blocking prerequisite for everything downstream in this dungeon. Context ≈ brief + cast + roster (~2k).
  2. Stage 2 — Draw: script, free. Reads that transcript once, emits (seed + window) bundles per section & tier. No agent.
  3. Stage 3 — Compose: fan out, parallel. The orchestrator (not the showrunner) spawns the fillers:
    • compose-short — one agent per section (~4–5/dungeon). Each emits quads covering that section's short tiers (~50/tier). Context = that section's seed bundles + specs.
    • compose-long — one agent per (section × long-tier) (high dungeons only). Each emits ~20–30 passages. Context = seeds + one long spec.
    • Where a dungeon has both, run compose-long first — its passages re-harvest (free, via a draw pass) into the short pools, cutting compose-short work.
  4. Stage 4 — Lint: script, free, per cell. Legality + dedup + coverage. On a shortfall, spawn one top-up filler for just the gap — never regenerate a passing cell. Then return the dungeon's pools.

~110 agent calls total (11 showrunners + ~80 fillers + ~20% top-ups). The runtime caps concurrency at min(16, cores−2) ≈ 10–14 live agents; everything else queues automatically, so the orchestrator can dispatch all 143 cells without minding the cap. The batch is token-budgeted so it hard-stops rather than overrunning.

Pilot first — the actual first action. Run only the Swamp end to end: 1 showrunner → draw → ~5 compose-short fillers → lint → ~2 top-ups ≈ ~8 agent calls. Record the real token spend, confirm the ~1M ×11 extrapolation holds, then release the other ten dungeons. Same measure-first discipline as the rest of this plan; it ships as a one-shot authoring run, never in the app.
