M0 Implementation Plan

The combat-math spike, built on a skeleton the rest of the game grows into.

M0's goal is unchanged from the roadmap: prove the combat math and monster typing feel right in a barebones page — winnable-but-tense in ~30–45s — before any art or animation. What this doc adds is how to build it so M1–M6 are additive, not a rewrite: the file layout, the state architecture, and a story-by-story task breakdown.

M0 is a spike, but not a throwaway. The UI is disposable; the engine/, config/, and content/ layers below are the real foundation the whole game keeps. The single rule that buys that: the combat engine never imports React.

Review notes folded into the plan

Four things surfaced reviewing the design docs; all are now reflected here and in the source docs.

#FindingResolution
AThe monster's time limit was undefined; reusing the player's short budget would leave slow monsters (Slime) unable to ever land a hit.Decided: monster budget = its own expected typing time × slack (~1.75). A per-battle tuning knob. See Monster typing.
B"15 chars/sec" for the player time limit ≈ 180 WPM (pro speed) — far too fast for a beginner 10-year-old.Fixed: it's a baseline typing speed, 15 WPM (≈ 1.25 chars/sec). Tuning it up/down is a core M0 job.
ClengthFactor could go to zero/negative for a very short prompt.Fixed: clamp with Math.max(0.25, …).
DThe monster pseudo-code's wait(interval) can't exist in single-threaded JS.Design: the engine is tick-driven (tick(dt)) with an injectable clock + seedable RNG, so the whole race is deterministic and testable.

State architecture: no Redux, three homes

The design doc's "React state, no external library" is the right call — but the reasoning matters, because following it too literally is where tech debt would actually accrue. A battle is a real-time simulation, not a form. Redux is not the fix; recognizing three kinds of state is.

KindExampleFrequencyHome
Simulation / battlemonster's live-typed string, both HP bars, both countdownsmany/secPlain-TS engine outside React; read via useSyncExternalStore
Persistent / savecoins, xp, skillTree, inventoryonce per battle/purchaseContext + useReducer M2
Ephemeral UIinput box text, current screenlocaluseState in the component
Context re-renders every consumer on every change — fine for a save file that changes once a battle, pathological for state ticking 60×/sec. That's the concrete reason the battle lives in an external-store engine rather than Context, and why Redux (which solves a different problem — predictable updates to sprawling shared trees) earns its keep in neither place. Keeping the engine React-free is also what makes the M0 deliverable — "prove the numbers" — testable headlessly (Story 2–4) and lets a balance simulator reuse it (Story 6).

File & component layout

The M0 skeleton. React-free layers (engine/, config/, content/, domain/) are the seam between combat logic and rendering — state/battle-store.ts is the only place the two meet.

Why this shape scales: adding the M2 save file is a new state/ context + a call site, not a combat-math change. Adding M6 sprites swaps MonsterTyping.tsx's renderer, not the sim. Adding a monster or a text tier stays a content edit. Every future milestone lands in exactly one layer.

Engine API sketch

// engine/battle.ts — a factory (no classes, per the code-style rules)
createBattle(config: BattleConfig) => {
  getState(): BattleState                     // authoritative snapshot
  subscribe(cb: () => void): () => void        // for useSyncExternalStore
  tick(dtMs: number): void                    // advance both cycles one frame
  submitPlayerAttack(input: string): void     // evaluate on Return (length-gated, exact match)
}

// A prompt source is sync-by-design: the tier's lines are loaded (possibly async)
// at battle SETUP, then handed to the engine as a plain picker so the sim stays pure.
type PromptSource = () => string

interface BattleConfig {
  combat: CombatConfig            // the tuning-knob object from config/combat.ts
  monster: Monster                // who we're fighting (from content/monsters.json)
  playerPrompts: PromptSource     // player's next-prompt source (skill-gated tier in M2)
  monsterPrompts: PromptSource    // monster's next-prompt source (its fixed text tier)
  rng: Rng                        // seedable; injected so a whole battle is reproducible
}

// BattleState (read by the UI)
{
  status: 'ongoing' | 'won' | 'lost'
  player:  { hp, maxHp, prompt, timeLimitMs, elapsedMs }
  monster: { id, hp, maxHp, prompt, typed, timeLimitMs, elapsedMs }
  lastEvent?: { side, kind: 'hit'|'miss'|'expire', damage? }   // for barebones feedback
}

The player's in-progress input stays UI-local (the human is doing the typing, not a simulation) and is handed to submitPlayerAttack(input) on Return. The engine owns the player's prompt and countdown so it can expire an attack and advance to the next prompt on its own.

Text content: data, not code

The content plan makes expandable text banks an explicit goal — "add ten lines to tier 3" should be a content edit, not a code change, and the kid should eventually be able to submit his own lines. A text-banks.ts full of string literals quietly breaks that. So text is treated the same way the SaveStore treats save data: data behind a provider interface, with the source swappable. 

// domain/types.ts
interface TextBank {
  loadTier(tier: TextTier): Promise<readonly string[]>   // async: allows lazy-loading later
  makePromptSource(tier: TextTier, rng: Rng): Promise<PromptSource>  // () => string, for the engine
}
ConcernLong-termM0 short-term (leads there)
Where text livesJSON files (tiers 1–6 hand-authored; 7–10 excerpted books), validated against a schemaPer-tier JSON under content/text/, imported by Vite
How it's readEverything goes through the TextBank interfaceOne bundled-JSON TextBank implementation
Large tiers (7–10)Swap the static import for a dynamic import()/fetch() so big book excerpts code-split and lazy-loadNot built — but the async loadTier shape is already there, so it's a localized swap
Kid's custom bankA second TextBank backed by IndexedDB, composed with the bundled oneNot built — composing two providers is a wrapper, not a rewrite
The async loadTier is deliberate even though M0's bundled JSON resolves instantly: it's the same lesson as choosing IndexedDB over localStorage — pick the async-shaped interface now so the future networked/lazy source is a swap behind one seam, not a rewrite of every call site. Crucially, loading is async but the engine only ever sees a synchronous PromptSource: a tier is loaded at battle setup, then the engine gets a plain () => string picker, so the combat sim stays pure and testable. (Monster stats follow the same "data behind a loader" idea via monsters.json.)

Keyboard visualization: sharing state with the prompt

An on-screen keyboard that highlights the next ~3 keys — brightest for the very next key, dimmer for the two after — directly serves the project's first principle ("typing quality first"): it teaches touch position (don't look at your hands) and lets the kid pre-load a multi-key sequence mentally. It's worth putting into the UI architecture from the start because it shares state with the prompt, and getting that seam right early avoids a refactor later.

The shared state is small and already exists — it's just placed at the right level:

  • The target prompt comes from useBattle() (engine state).
  • The input (what's typed so far) is the UI-local state from Story 5.
  • Both PlayerPrompt and Keyboard need both values, so input lifts to their nearest common parent, BattleScreen, which passes prompt + input down to each. Standard "lift state up" — no context needed at M0 scale; it can graduate to a scoped TypingContext only if the tree deepens.

Highlighting is a function of the actual input, not just the cursor position — because a mistake changes what the correct next keystroke is. Three cases:

  • On track (input === prompt.slice(0, input.length)) → highlight the next ~3 keys in descending brightness.
  • Off track (a wrong character anywhere in what's typed, or over-typed past the end of the prompt) → highlight only Backspace. Backspacing is the single correct move until the input is a valid prefix again, so nothing else should glow.
  • Complete & correct (input === prompt) → highlight Enter, the key that submits the attack.
This is why M0 needs the full five rows, not just the three letter rows: Backspace lives on the number row and Enter on the home row, so both must render from the start. The layout data therefore includes the number row, the Tab/Caps/Shift modifiers, and the space bar, and Key.tsx handles variable key widths — even though M0 only ever highlights letters, space, Backspace, and Enter. 
// ui/battle/keyboard.ts — pure, unit-testable, no React
keyForChar(char: string): { code: string; shift: boolean }   // 'A' → {code:'a', shift:true}, ' ' → 'Space'
nextKeyHighlights(prompt: string, input: string, n = 3): Map<string, number>
//   on track  → { <next code>: 0, <+1>: 1, <+2>: 2 }   (0 = brightest; nearest wins on duplicates)
//   off track → { 'Backspace': 0 }
//   complete  → { 'Enter': 0 }
The keyboard is pure derivation: given prompt + input it computes which keys glow and holds no state of its own. That keeps nextKeyHighlights testable outside React, keeps the layout swappable keyboard-layout.ts data, and makes keyForChar the single seam where tiers 4+ (capitals, punctuation, symbols) teach themselves — highlighting the base key plus Shift — without touching battle logic. M0 ships the letters/space/Backspace/Enter cases; the seam is ready for the rest.

Stories & tasks

Build order: 0 → 1 → 2 → 3 → 4 → 6 → 5 → 7. Note Story 6 (the balance harness) lands before the UI — prove the numbers headlessly, then build a throwaway-grade UI to feel them. That's the most faithful reading of "prove the numbers before investing in any polish." Story 7 (keyboard) follows Story 5 because it consumes the shared prompt/input state that Story 5 establishes.

Story 0 — Scaffold & tooling foundation

Stand up the project and the code-style baseline, minus the deploy pieces (those are M4).

  • Vite react-ts template; add Tailwind.
  • ESLint airbnb + airbnb-typescript + react-hooks; Prettier (printWidth:100, semi:false, tabWidth:2) + eslint-config-prettier.
  • The two no-restricted-syntax bans (no function, no class → factory functions); relax no-console to warn.
  • Vitest + @testing-library/react.
  • Create the folder skeleton with placeholder barrels.
  • npm scripts:
    • lint → eslint .
    • lint:fix → eslint . --fix
    • build → tsc -b && vite build (type-checks and produces the optimized prod build)
    • test → vitest run
    • presubmit → npm run lint && npm run build && npm run test (the gate to run before every commit)
    • M4 deploy — added in the ship-it milestone once hosting exists
  • defer to M4 husky + lint-staged pre-commit (wire it to presubmit) and the GitHub Actions lint/build gate.

Done when: npm run dev shows a page; lint and test run green on an empty suite.

Story 1 — Domain types, config, content foundation

  • domain/types.ts: Monster, TextTier, BattleConfig, PromptSource, TextBank, Rng, PlayerState, MonsterState, BattleState, DamageResult.
  • config/combat.ts: every knob in one exported object — baseDamage, referenceLength, lengthFactorFloor: 0.25 (finding C), playerBaselineWpm: 15 (finding B), avgWordLength: 5, monsterSlack: 1.75 (finding A), crit defaults, typing variance.
  • Content as data (see Text: data, not code): content/text/tier-01.json + tier-02.json; content/text-banks.ts as the thin bundled-JSON TextBank provider; content/monsters.json (Slime + Goblin) with a typed loader; content/keyboard-layout.ts (QWERTY rows).

Done when: types compile; content loads through the TextBank interface (no string literals in .ts); config imports cleanly with zero logic in it.

Story 2 — RNG + damage math core M0 deliverable

  • engine/rng.ts: createRng(seed) (mulberry32) with next() → [0,1) and sample(mean, variance).
  • engine/damage.ts: lengthFactor (clamped), speedBonus(timeUsed, timeLimit), critMultiplier(luck, rng), computeDamage(…).
  • Tests: table-driven cases; short-prompt clamp; speed-bonus boundaries; crit determinism under a fixed seed.

Done when: damage is a pure function of inputs + seed, fully covered.

Story 3 — Monster typing simulation core M0 deliverable

  • engine/monster-typing.ts: createMonsterTyper(monster, prompt, rng) with advance(dtMs) — accumulates time toward the next (sampled-interval) character, rolls accuracy for typos, models the time-based notice/correction (finding D). Exposes { typed, done, failed, elapsedMs }.
  • Tests (seeded RNG + manual dt stepping): a high-accuracy monster converges to the target string; a low-accuracy/slow-attention one visibly overshoots then corrects; a tight slack makes it blow its budget.

Done when: stepping advance() reproduces identical typed-strings for a given seed.

Story 4 — Battle engine core M0 deliverable

  • engine/battle.ts: createBattle(config) integrating the player cycle (prompt → countdown → submitPlayerAttack(input) → hit/miss/expire → next prompt) and the monster cycle (drive monster-typing, resolve on completion, start next), HP, and win/lose. getState/subscribe/tick(dt).
  • Honor the submit rules: Return only counts once input length matches the target; exact match → hit, else miss; no re-attempt.
  • Tests: a scripted battle (seeded RNG + a list of (timeOffset, playerInput) submissions) produces a deterministic winner and duration.

Done when: a headless battle runs start → finish deterministically.

Story 6 — Tuning / balance harness makes "done" defensible

This is what lets you answer M0's question instead of guessing at it.

  • engine/sim/balance.ts: a Node/vitest script that runs N headless battles with a simulated player of configurable wpm/accuracy vs. a monster, reporting win rate + median duration. ~30 lines, because the engine is React-free.
  • Use it to land playerBaselineWpm, baseDamage, and the monster stats on "winnable-but-tense, ~30–45s" before touching UI.

Done when: you can answer "what win rate does a 20-wpm kid get vs the Slime, and how long does it take?" without playing 50 games by hand.

Story 5 — React bridge + barebones UI playable: barebones battle

  • state/battle-store.ts (useSyncExternalStore adapter), ui/hooks/useGameLoop.ts (rAF → tick, stop on end), ui/hooks/useBattle.ts.
  • BattleScreen composing HealthBar ×2, MonsterTyping, PlayerPrompt, BattleResult. Deliberately unstyled — Tailwind defaults only.
  • Input is local useState; Enter → actions.submit(input) once length matches.

Done when: the literal M0 deliverable — type a line, see damage land, watch the monster's bar tick down on its own clock.

Story 7 — Keyboard visualization (initial) playable: type-without-looking aid

An initial on-screen keyboard — planned now so the shared prompt/input seam is right from the start, even though the render is minimal.

  • Lift input from PlayerPrompt up to BattleScreen; pass prompt + input to both PlayerPrompt and Keyboard.
  • content/keyboard-layout.ts: the full 5-row QWERTY as data — number row (with Backspace), the letter rows, Enter, the modifier keys, and the space bar.
  • ui/battle/keyboard.ts: pure keyForChar() and nextKeyHighlights(prompt, input) implementing the three cases (on-track → next ~3 keys; off-track → Backspace only; complete → Enter). M0 covers letters + space; the Shift seam is stubbed for later tiers.
  • Keyboard.tsx / Key.tsx: render all five rows with variable key widths; highlighted keys glow in descending brightness (nearest = brightest).
  • Tests for nextKeyHighlights(): on-track at start/mid (incl. duplicate-key-in-window, nearest wins); one wrong char anywhere → only Backspace; over-typed past the prompt → only Backspace; fully-correct input → only Enter.

Done when: as he types, the coming keys light up in order; the instant he mistypes, everything goes dark except Backspace until he fixes it; and Enter lights up once the line is correct and complete.
