Codebase Architecture

A living map of the src/ tree — the layering, the state homes, the seams, and where each system lives. Read this before spelunking the code.

This is orientation, not gospel: the code is the source of truth, and this doc will drift. Its job is to make the next read cheap — to say "here is the shape, here are the rules, here is which file to open" so a change starts from a map instead of a search. Keep it roughly current when the shape changes; don't sweat line-level detail. Snapshot re-taken post-M3 (the D&D character layer has shipped).

Stack

  • React 19 + TypeScript, built with Vite 8, styled with Tailwind 4 (design tokens in an @theme block; see tokens.json).
  • IndexedDB via idb for the save file — no backend, no accounts (the brief's "one kid, one Chromebook" rule).
  • Vitest 4 (jsdom, globals) for tests; @testing-library/react + user-event for components, fake-indexeddb for storage.
  • No router, no Zustand/Redux. State is React Context + useReducer plus one hand-rolled external store for the battle. Deliberate — see state homes.

The layering & its one rule

The tree is organized by layer, not by feature, and the layers form a one-way dependency stack. The React-free layers (domain/, engine/, config/, content/) never import React or the UI; the state/ layer is where logic first meets React; ui/ renders. The load-bearing invariant that keeps it all decoupled:

The combat engine/ never imports React. The persistent save never imports combat, and combat never imports the save. They meet at exactly one pure function — resolveModifiers() — which turns save data into a PlayerModifiers object the battle consumes. This single seam is why milestones stay additive: M3 swapped what feeds it (a Character + equipped Weapon + active consumable buffs, replacing M2's skill tree) without touching the combat math on the other side. Grow the seam; don't route around it.

State: three lifetimes, five stores, no global library

Every piece of state is classified by lifetime, and that decides its home. Conflating lifetimes is the bug class this structure exists to prevent (e.g. persisting a dungeon run would let a player quit-to-dodge a wipe).

KindExampleLifetimeHome
Simulation / battleboth HP bars, both countdowns, the monster's live-typed string one fight state/battle-store.ts — a hand-rolled external store wrapping one createBattle(), read via useSyncExternalStore in ui/hooks/useBattle.ts; useGameLoop drives tick
Persistent / savethe character (class/level/XP/abilities), coins, equipped weapon + owned inventory, hearts.max, highestUnlockedTier, stats forever (survives reload) state/save/ — Context (SaveProvider/useSave) + useReducer, mirrored to IndexedDB (debounced)
Ephemeral / runthe generated graph, which nodes are cleared, hearts remaining, active consumable buffs one dungeon visit (discarded on death/leave/reload) state/dungeon-run/ — a second Context + useReducer, scoped under the dungeon screen, never touched by IndexedDB
Ephemeral / UIinput box text, which screen is upa render local useState; navigation is useState<Screen> in GameShell
The hearts split is the canonical example of why lifetimes matter: hearts.max is a save fact (earned, permanent), while hearts remaining is a run fact (starts at max, spent on losses, gone at run end). Keeping "remaining" in the ephemeral store makes "close mid-dungeon = leave" fall out for free. M3 reuses this exact split for consumables — owned items are persistent, active buffs are ephemeral.

Navigation: a screen state machine, not a router

app/navigation.ts defines Screen = { name:'map' } | { name:'inn' } | { name:'dungeon'; tier } | { name:'shop' } with constructor helpers (toMap/toInn/toDungeon/toShop). app/GameShell.tsx holds one useState<Screen>, switches on screen.name, and — ahead of any screen switch — gates on save.character === null: a fresh or freshly-migrated save renders CharacterCreateScreen instead (not a Screen variant itself — there's nowhere to navigate "back" to from it) until a character exists. Battle is not a top-level screen either — it's launched inside DungeonScreen so the ephemeral run stays mounted, and returns its result via an onResult('win'|'lose') callback. The Shop is now a real screen (M3 Story 10), reachable from the map's Shop HubCard. Four Screen variants plus the pre-map creation gate, for one kid on one device, is why there's still no router library.

The combat engine (where the math lives)

  • engine/battle.ts — createBattle(config) returns { getState, subscribe, tick(dtMs), submitPlayerAttack(input) }. Runs the player and monster as independent cycles on one clock. A player submit only counts when input.length === prompt.length (exact-match). Player time limit = max(floorMs, readingBufferMs + expectedTypingTimeMs(len, baselineWpm)).
  • engine/damage.ts — all core math: lengthFactor (longer = harder), speedBonus (fast finish = up to 2×), rollIsCrit, tierGatePenalty (squared tier-ratio when served below the fight's target tier), and computeDamage = (sum of weaponDie rolls + weaponAbilityMod) × damageScale × lengthFactor × speedBonus × powerUpMultiplier × tierGatePenalty × fumbleDamageMultiplier — the M3 Story 7 rework: a flat baseDamage became a rolled weapon die + ability mod, and a crit rolls critCount dice (2 normally, 3 for the Wizard) and sums them instead of doubling a flat result. computeMonsterDamage stays the old flat shape (baseDamage × lengthFactor × speedBonus, no dice/crit/power-ups, all player-only). Now lit: powerUpMultiplier (driven by consumables like Bull's Strength) is finally read by computeDamage. Still a gap: DEX's critChanceBonus, a weapon's critRange, and a crit-boost item's critDamageMult are all computed into PlayerModifiers (engine/character/ modifiers.ts) but rollIsCrit/computeDamage never read them — crit chance today is still the flat combat.criticalChance, unchanged since M0. A known pending wiring gap (a tuning-coupled pass, not an oversight), not yet scheduled to a story.
  • engine/monster-typing.ts — the monster typer: chars arrive on a noisy interval, each may be a typo per accuracy, typos aren't corrected until attention seconds pass; timeLimitMs = expectedTypingTimeMs(...) × slack.
  • engine/rng.ts — createRng(seed), mulberry32, deterministic; next() uniform, sample(mean, variance) normal.
config/combat.ts holds the committed combat constants (baseDamage: 10, playerMaxHp: 40, criticalChance: 0.1, …). Per the working rules it stays at its committed values — retuning happens by changing config, and any change is a deliberate, reviewed act, not a drive-by.

The one seam: resolveModifiers()

engine/character/modifiers.ts exports resolveModifiers(character, weapon, activeBuffs?, cfg?): PlayerModifiers — its new M3 home, replacing the retired engine/progression/skill-effects.ts. It folds a persistent Character's level/abilities, the equipped Weapon's die/ability/crit range, the character's class feature, and the run's ephemeral ActiveBuff[] into one PlayerModifiers bag — maxHp/maxHearts (level + CON), intTierCap (INT), the encounter d20's encounterBonus/hasAdvantage, DEX's critChanceBonus/dodgeChance, CHA's intimidateWpmCut, the weapon's weaponDie/weaponAbilityMod/ critRange, and class-feature flags (guaranteedFirstCrit, sneakAttackDice, secondWind, arcaneCritMult). It is the sole bridge from save data to combat numbers, called by battle-store.ts (which then builds the fight) and by the map/dungeon/Inn screens (for maxHearts/maxHp). Still the highest-leverage function in the codebase: keep its output contract and you can rewrite everything feeding it without touching the engine — M3 already did exactly that once (see M3 seams).

The save file

domain/save.ts defines SaveData (now version: 3) — character (null until creation), coins, equippedWeapon, inventory: { weapons, consumables }, hearts.max, highestUnlockedTier, monstersDefeated, stats. No more xp/skillTree at this level — XP now lives on character.xp. isSaveData() hard-requires version === 3; migrate() knows one real migration (a v2 save: keeps coins/tier-unlock/defeated-list/stats, drops skillTree, sets character: null so the player is routed through character creation once) and falls back to a fresh default for anything older/unrecognized. state/save/: save-reducer.ts (pure; actions include createCharacter, gainXp, applyAsi, equipWeapon, buyWeapon, buyItem, consumeItem, unlockTier, recordDefeat, hydrate — costs are passed in so the reducer never imports config), save-storage.ts (IndexedDB, 250ms debounced write), SaveProvider.tsx (hydrates on mount with a 4s timeout fallback, mirrors every change). Now lit (M3): stats.bestWpm is written on every win (a personal-best floor, never lowered). Still dead: stats.battlesLost — no action increments it yet.

Content & the key types

  • Monsters — Monster { id, name, tier, role('regular'|'boss'|'mimic'), habitat, hp, textTier, wpm, accuracy, attention, slack, flavor }; 70 records in content/monsters.json, queried via getMonster/byTier/byRole. See also monster-manual.json for canonical names/CRs.
  • Dungeons — DungeonGraph { tier, seed, entranceId, bossId, waypointId, approachId, nodes }; nodes are { id, kind, state, monsterId?, edges, isRealChest? }. generateDungeon() builds a two-fan-out graph; graph.ts has the pure transitions (clearNode, availableNodes, bossUnlocked, isComplete). Config: DungeonTier { tier, habitat, wpmRange, textTierRange, rosterPool } — textTierRange was dead through M2; engine/dice/band.ts's bandToServedTier() is now its load-bearing consumer, turning the encounter d20's band into this fight's served text tier.
  • Text tiers — TextTier = 1..10; TextBank { loadTier, makePromptSource } with a degrade-to-nearest-easier fallback and a never-repeat-the-last-line picker; banks in content/text/tier-0N.json.

Tests

Vitest, configured inside vite.config.ts (jsdom, globals, src/test/setup.ts). Tests are co-located as *.test.ts(x) siblings (the engine uses a __tests__/ subfolder). Well covered: the engine (battle, damage, monster-typing, rng, the character seam/leveling/ ability-roll/pricing, the encounter/band/mimic-sense dice), the dungeon (graph + invariant sweep), progression (rewards), the balance sim + combat-invariants sweep, content (monsters, text-banks), state (save-reducer, save-storage, SaveProvider, dungeon-run-reducer), and a growing set of UI units (PlayerPrompt, TypedProgress, graph-layout, RewardModal), plus battle-store itself. Not covered: the hooks and most top-level screens. The habit: engine/data logic is proven headlessly (unit tests + the engine/sim/ harnesses) before the UI that consumes it — mirror it.
